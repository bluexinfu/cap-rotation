# 開發流程與範本

> **這份文件的角色**：定義一個改動從「想法」走到「上線」要經過哪幾步，並提供可直接複製的範本（變更提案、ADR、發版檢查清單）。單人版，刻意精簡。

- 最後更新：2026-06-16
- 維護者：單人

---

## 1. 改動的生命週期

依改動大小走「快慢兩條道」。判斷標準：**會不會影響別人對這張圖的解讀，或改變產品邊界**。

```
                 ┌─ 小改動（bug、文字、視覺微調）
   想法 ─判斷大小┤      └─▶ 直接做 ─▶ 自我驗收(DoD) ─▶ 記 CHANGELOG ─▶ 完成
                 │
                 └─ 大改動（新功能、改指標、改 schema、動非目標）
                        └─▶ ① 對照定位 ─▶ ② 寫提案/ADR ─▶ ③ 更新 PRD
                               ─▶ ④ 實作 ─▶ ⑤ 自我驗收(DoD) ─▶ ⑥ 記 CHANGELOG ─▶ ⑦ 發版
```

### 小改動（快道）
直接改 → 用 [DoD](02-change-governance.md#5-完成的定義definition-of-done) 自我驗收 → 在 [CHANGELOG](../CHANGELOG.md) `[Unreleased]` 記一行。完成。

### 大改動（慢道）
1. **對照定位**：回 [產品定位](00-product-brief.md) 確認它符合方向、沒踩[非目標](00-product-brief.md#6-非目標明確不做的事)。不符合就停（或寫 ADR 推翻非目標）。
2. **寫提案 / ADR**：用 §3 範本想清楚「為什麼、怎麼做、有什麼後果」。涉及岔路取捨的，這份提案就是 ADR。
3. **更新 PRD**：在 [PRD](01-prd.md) 新增/修改功能條目與驗收標準。
4. **實作**。
5. **自我驗收**：逐項過 [DoD](02-change-governance.md#5-完成的定義definition-of-done)。
6. **記 CHANGELOG**。
7. **發版**：用 §4 發版檢查清單。

> 單人專案沒有「審核者」，但「對照定位」這一步等於請過去的你（定位文件）來審核現在的你。這是這套機制最值錢的地方。

## 2. 怎麼決定優先級

待辦放在 [ROADMAP](ROADMAP.md)，用兩個維度排：

- **價值**：對核心 use case（尤其 UC1 掃描、UC3 預警）的幫助有多大？
- **正確性**：它是不是在修「會誤導使用者」的問題？

**修「誤導」永遠優先於加「新功能」**——呼應[設計原則 2：誠實的不確定性](00-product-brief.md#5-設計原則取捨時的優先序)。例如 v0.2.0 把「時間基準 bug」與「相對強度標準化」一起處理，但前者（消除自我矛盾）的優先級高於後者（加新視角）。

## 3. 範本

### 3.1 CHANGELOG 條目範本

```
### Fixed
- 一句話描述改了什麼（對應功能編號 F-XX；必要時 見 ADR-XXXX）
```
動詞開頭、結果導向。讓三個月後的你掃一眼就懂。

---

### 3.2 變更提案 / ADR 範本

複製到 `decisions/ADR-NNNN-短標題.md`：

```markdown
# ADR-NNNN：<決定的標題>

- 狀態：提議中 | 已採納 | 已棄用 | 被 ADR-XXXX 取代
- 日期：YYYY-MM-DD
- 相關：F-XX（PRD 功能）、其他 ADR

## 背景
要解決什麼問題？為什麼現在要決定？當下的限制與事實是什麼？

## 決定
我們決定怎麼做（一句話講清楚），以及關鍵的實作方向。

## 理由
為什麼選這個？它怎麼解掉背景裡的問題？

## 後果
- 好處：
- 代價 / 風險：
- 需要連帶改的東西（PRD？schema？文件？）：

## 考慮過的替代方案
- 方案 B：…（為什麼沒選）
- 方案 C：…（為什麼沒選）
```

---

### 3.3 發版檢查清單

每次發版複製一份逐項勾：

```markdown
## 發版 vX.Y.Z — YYYY-MM-DD

- [ ] 所有改動都過了 DoD（docs/02 §5）
- [ ] 決定版本號層級（MAJOR/MINOR/PATCH，依 docs/02 §2）
- [ ] CHANGELOG 的 [Unreleased] 改名為 [X.Y.Z] - YYYY-MM-DD，並開新的空 [Unreleased]
- [ ] 重大決定都有對應 ADR
- [ ] PRD 與功能現況一致
- [ ] 前端在 file:// 與 HTTP 部署兩情境都驗過
- [ ] build_data.py 能跑出合法 data.json（若有改資料層）
- [ ] (已接 git 時) git tag vX.Y.Z 並 push
- [ ] 部署到 GitHub Pages 後實際開網址確認
```
