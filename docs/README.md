# 資金輪動 · 專案治理總覽

> 這個資料夾是「資金輪動」這個產品的大腦：它記錄**為什麼做、做了什麼、怎麼演進**。
> 程式碼會告訴你「現在長怎樣」，這裡的文件告訴你「為什麼長這樣、接下來往哪走」。

維護者：單人。所以這套機制刻意走**輕量但有紀律**：不開會、不寫一堆表單，但每個重要決定與每次改動都留得下痕跡，讓「三個月後的你」看得懂、追得回。

---

## 這套治理體系長怎樣

四份核心文件 + 一套流程，彼此互相扣：

```
產品定位 (00)  ──定方向──▶  PRD (01)  ──拆功能──▶  程式碼 (index.html / build_data.py)
     │                          │                          │
     │                          │                          │ 每次改動
     ▼                          ▼                          ▼
為什麼存在、給誰、         功能清單、use case、        CHANGELOG (變更紀錄)
背後的 RRG 理論           驗收標準、追溯矩陣          + 決策紀錄 ADR (為什麼這樣改)
                                                          │
                                                          ▼
                                          流程與範本 (03)：怎麼從想法 → 決策 → 上線
```

| 文件 | 回答的問題 | 多久看一次 |
|---|---|---|
| [`00-product-brief.md`](00-product-brief.md) 產品定位 | 這是什麼？給誰？為什麼值得做？背後的理論是什麼？ | 方向有疑問、或要拒絕一個功能時 |
| [`01-prd.md`](01-prd.md) 產品規格 | 有哪些功能？對應什麼使用情境？怎麼算「做完」？ | 要加/改功能前 |
| [`02-change-governance.md`](02-change-governance.md) 變更治理 | 版本怎麼編？變更怎麼記？決策怎麼留？ | 每次發版時 |
| [`03-workflow-and-templates.md`](03-workflow-and-templates.md) 流程與範本 | 一個改動要走哪幾步？有哪些範本可抄？ | 開始一個新改動時 |
| [`ROADMAP.md`](ROADMAP.md) 待辦藍圖 | 接下來要做什麼？優先級？ | 決定下一步做什麼時 |
| [`../CHANGELOG.md`](../CHANGELOG.md) 變更紀錄 | 哪一版改了什麼？ | 每次發版、或回溯問題時 |

## 快速上手：我現在想做某件事，該翻哪裡？

- **想加一個新功能** → 先看 [`00`](00-product-brief.md) 確認它符合產品定位（不符合就拒絕）→ 在 [`01-prd.md`](01-prd.md) 補一條功能規格 → 照 [`03`](03-workflow-and-templates.md) 的流程做 → 改完更新 [`CHANGELOG`](../CHANGELOG.md)。
- **要修一個 bug** → 直接做，改完在 [`CHANGELOG`](../CHANGELOG.md) 的 `Fixed` 記一筆。
- **做了一個會影響未來的重大設計決定**（例如改演算法、換資料來源）→ 在 [`decisions/`](decisions/) 寫一份 ADR。
- **要發布一個新版本** → 照 [`02`](02-change-governance.md) 的版本規則編號、整理 CHANGELOG、跑發版檢查清單。
- **忘了某個東西當初為什麼這樣設計** → 翻 [`decisions/`](decisions/) 和 [`00`](00-product-brief.md)。

## 為什麼單人專案也值得做這個？

因為「未來的你」其實是個陌生人。三個月後你會忘記今天為什麼把縱軸叫「動能」而不是「加速度」、為什麼用絕對金額而不是標準化、為什麼選 5 日視窗。沒有紀錄，你會重複踩同樣的坑、或無謂地推翻一個其實有道理的舊決定。

這套機制的成本被刻意壓到最低：大多數時候你只需要在 CHANGELOG 加一行；只有真正的「岔路決定」才需要寫一份 ADR。
