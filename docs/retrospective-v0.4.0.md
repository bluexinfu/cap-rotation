# 開發過程回顧 · v0.4.0

> 這份是 v0.4.0 整段開發的**敘事旅程＋方法論心得**，把散在 CHANGELOG / ADR / commit 裡的點串成一個故事，供日後回顧。日期：2026-06-27。

## 緣起：一個「週五沒更新」的小狀況

起點不是計畫中的功能，而是 Ted 發現 **6/26（週五）盤後資料沒更新**。一查：GitHub Actions 有跑、有 commit，但 `data.json` 停在 6/25。根因是 **TWSE 的 T86（三大法人）端點延遲出表**——建置當下抓不到 6/26，而 `build_data.py` 設計成「沒有 T86 就跳過整天」。關鍵發現：**FinMind 其實已經有 6/26 的法人資料**，代表資料存在、只是 TWSE 那個端點慢。

## 第一階段：救火 + 治理

- **資料韌性**：新增 `fetch_t86()`——先打 TWSE，缺資料時改用 **FinMind 全市場備援**重建（需 `FINMIND_TOKEN`）；排程由單一 18:30 擴成 **18:30／21:00／隔日 07:00 三時段**；commit 訊息改用實際最新交易日。後來再加 **R10 更新失敗通知**（停擺會 workflow 失敗→自動寄信）。
- **治理原則確立**：本機 clone 早已落後又夾雜未存的改動（差點弄丟 R13~R15 待辦）。於是**封存本機副本、確立「改碼一律走雲端 GitHub」**，並把每條改動的「為什麼」留 ADR、「使用者白話」留進 app。

## 第二階段：方法論升級（自身波動 σ 三部曲）

- **R8 · 中性門檻改各板塊 σ**（ADR-0008）：資料探索發現固定 ±3 億門檻**對不同體型板塊極不公平**（半導體 1% 時間中性、造紙 100%）。改 1σ 後：公平性離散度 36→7、分類翻轉率 14.4%→6.6%。**先用 70 日真實資料試、不是憑公式想當然**。
- **R14 · 逆勢強勢加 1σ 門檻**（ADR-0009）：**源自 Ted 的一句觀察**——「為什麼造紙只有 0.3σ 也上金？」。查出金色是「逆勢強勢」高亮、沒有最低門檻，弱盤裡「矮子裡的高個」也上金。加 1σ 門檻（與 R8 對齊），實測 6/15 由 5 個收斂到 2 個真強勢。**使用者觀察直接變成修正。**
- **R13 · 市場相對第 3 座標**（ADR-0010）：設計選擇「扣大盤潮水＋除自身 σ」。**資料探索抓到一個會壞掉的坑**：直覺的「(x−大盤潮水)÷σ」會讓小板塊爆表到 −11σ（因為平均潮水對小板塊σ是天文數字）。**改順序**為「先換 σ 強度、再扣平均」就正常了（範圍 −1.2σ~+2.4σ）。

## 方法論心得（這次最值得留下的）

1. **資料探索先行**：兩個「會壞掉」的設計（市場相對 −11σ 爆表、固定門檻不公平）都是**真實資料試出來、不是公式階段看得出來的**。
2. **每條改動走完整一圈**：探索 → ADR（為什麼）→ 實作 → 使用者白話說明 → CHANGELOG/ROADMAP → 線上驗收。
3. **使用者的一眼觀察 = 最好的 bug 來源**（R14、播放閃動、顏色說明、版本號都是）。
4. **治理**：單一事實來源在雲端；ADR 留 why、in-app 留使用者版；commit 前給看。
5. **守住非目標**：σ 門檻是「觀察用的尺」，**不是進場訊號**；越調敏感、雜訊越多。門檻只管顏色、不管位置，早期跡象看「位置漂移＋動能＋累積」即可。
